# 小規模チームの開発フロー

## 基本フロー

```text
Issue作成
→ 作業ブランチ
→ Codexまたは手作業で変更
→ typecheck / lint / test / build
→ Pull Request
→ 別メンバー確認
→ mainへ統合
```

## Issue

Issueへ次を残します。

- 背景と利用者への価値
- 今回の目的、対象外、完了条件
- Codexへ渡した依頼内容と重要な制約
- データ送信、互換性、プライバシー上の懸念
- 手動確認が必要な項目

1つのIssueへ複数の無関係な機能を混ぜません。破壊的変更、データ移行、API／有料サービス追加は作業前に相談します。

## ブランチと作業

- mainへ大きな変更を直接入れない。
- `feature/<短い目的>`、`fix/<短い目的>`、`docs/<短い目的>`など、内容が分かる作業ブランチを使う。
- 作業開始前にmainの状態とテスト結果を確認する。
- 録音、文字起こし、要約、`.env`、APIキー、モデルをコミットしない。
- AI生成コードも人間が読み、設計・安全性・ライセンス・テストを確認する。

## 変更後の確認

```powershell
pnpm.cmd run typecheck
pnpm.cmd run lint
pnpm.cmd test
pnpm.cmd run build
powershell -ExecutionPolicy Bypass -File scripts\check-share-readiness.ps1
```

レビュー依頼前に`git diff`、`git status`、可能なら`git diff --check`を確認します。生成物や実データが混ざっていないことを、ファイル名だけでなく差分内容でも確認します。

## Pull Request

Pull Requestへ次を書きます。

- 変更した理由と利用者への影響
- 主な変更点と変更しなかった内部処理
- typecheck／lint／test／buildの結果とテスト件数
- 実ブラウザー、実マイク、実APIなどの実施／未実施
- 秘密情報・録音・モデルを含まないこと
- 既知の制限、リスク、戻し方

## レビュー

別メンバーが最低限次を確認します。

- Issueの目的外の変更が混ざっていない
- rawText、録音、接続、IndexedDBの既存安全境界を壊していない
- 外部送信が増えていない、または明示的に合意済み
- ログへ本文・秘密値を出していない
- `.gitignore`だけに頼らず、差分に危険ファイルがない
- テストが意味のある失敗条件を含む

承認後にmainへ統合します。強制pushや履歴書き換えは通常フローに含めません。

## 所有権とライセンス

LICENSEはまだ選択していません。共有開始前に次を決めます。

- リポジトリを団体所有にするか個人所有にするか
- 非公開開発におけるコードの所有権と利用範囲
- 一般公開時のライセンス選択と承認者
- 外部ライブラリのライセンス確認方法
- AI生成コードを含む全変更のレビュー責任

特定のライセンスを自動的に採用しません。
